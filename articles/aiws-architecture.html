把多智能体协作做进企业真实工作流：AIWS 的架构与取舍

过去一年，"AI Agent" 几乎成了每个团队的必选项。但大多数落地，最后都长成同一个样子：一个独立的聊天框，问它点什么、它答点什么，然后——就没有然后了。

我在做的 AIWS（智能组织协作平台）想解决的，恰恰是这个"然后"：让多智能体协作真正嵌入企业的日常工作流，而不是停在一个聊天窗口里。这篇文章讲我们的架构选择，以及更重要的——那些"为什么这么选"的取舍。

一、为什么"又一个 AI Chat"没用

先说真问题。组织内部每天都在产生大量讨论：群里对齐需求、会上拍板、临时协商。但这些讨论沉淀成可执行任务的成本极高——讨论完了，谁来建任务、谁来跟进、状态同步给谁，大量靠人肉。

而市面上的 AI 助手，普遍停在"独立入口"：

  • 它不知道组织里有谁、谁负责什么；
  • 它不能真的去建一个任务、指派给人、追踪到结束；
  • 它给的建议没人能确认、也没法追溯。

换句话说，它没有进入真实的协作与执行链路。要让 AI 有用，它必须能"做事"，而且做的事可理解、可确认、可执行、可追踪——尤其在严肃的组织/业务场景里，这四个词是底线。

二、目标与约束

AIWS 的设计目标，可以浓缩成一句话：

把 AI 从"会聊天"变成"能在组织里可控地做事"。

由此推出几条硬约束：

  1. 不能失控：AI 可以建议、可以起草，但关键动作（建任务、指派、对外）必须有人确认。
  2. 要有组织上下文：它得知道人、角色、任务、历史，否则建议是空的。
  3. 要可追溯：每一步 AI 行为都要能回溯"它为什么这么做、谁确认的"。

三、整体架构

技术栈（真实）：

  • 前端：React + TypeScript + Vite + TailwindCSS
  • 后端：Python + FastAPI + SQLAlchemy
  • AI 层：AgentScope + A2A（Agent-to-Agent）多智能体协议 + OpenAI 兼容 API
  • 数据：MySQL
  • 集成：企业微信（身份、消息、协作入口）
  • 实时：SSE 事件推送

一个简化的分层视图：

企业微信 / Web 前端（协作入口）
↓
FastAPI 网关　认证 / 路由 / SSE
↓
SuperAgent
意图理解 / 路由 / 汇总
业务线自助配置的智能体
各带知识库，后台自动生成"能力概要"
工具层

SuperAgent 按概要匹配 → 分发给最相关 ≤2 个 → 并行作答 → 汇总（A2A 协议）

核心是中间这层：业务线自助配置带知识库的智能体，后台为每个智能体自动生成能力概要；SuperAgent 据此做动态路由——用户未指定时分发给最相关的最多两个智能体并行作答，再汇总返回。所有"会改变世界"的动作，都要穿过 Human-in-the-loop 这道闸。

四、四个关键设计决策

这部分是重点——架构图谁都会画，真正体现工程判断的是这几个"为什么"。

1. 嵌入 IM（企业微信）：让 AI 待在"工作真正发生的地方"

要让 AI"进入真实工作流"，第一步就是进入 IM——因为组织里的讨论、对齐、临时协商，大量就发生在企业微信的群聊里。AI 如果不在那里，它永远是"另开一个窗口"的外人。

所以我们把企业微信作为第一协作入口，而不是又做一个独立 App：

  • 身份与组织直接复用企业微信，AI 天然知道"谁在说话、属于哪个团队"；
  • 在群里 @ 智能体即可发起请求，AI 自动读取 @ 之前最近一段群聊消息作为上下文——"把刚才那段讨论拆成任务"才有真正的"刚才"，不必让用户复述背景；
  • 把"刚才那段讨论"直接起草成任务（草稿），人确认后建任务，并把进展回推到 IM 追踪；
  • 确认卡片、任务进展、通知通过 SSE / 企业微信消息实时回推，形成"讨论 → 任务 → 跟进"的闭环，而不是把人甩去另一个系统。

更进一步：群聊里的讨论不会随聊天流走——它们会由 LLM 自动提炼成要点 / 问答对，沉淀进知识库（顺手做了去噪和结构化，而非整段照搬）。一次性的对话因此变成可复用的组织知识，再反哺给智能体（见下一节）——讨论越多，智能体越懂这个组织。于是 IM 既是协作入口，也是知识的来源，和"智能体带知识库"接成一个数据飞轮：

群聊讨论
 →
知识库
 →
智能体能力概要
↺　智能体回答更懂组织、更准 → 又激发更多讨论，闭环反哺

一句话：让 AI 待在工作真正发生的地方，而不是把人拉到 AI 待的地方。这是"嵌入工作流"和"又一个聊天框"最本质的区别。

2. 多智能体协作：不预先划分边界，而是"业务线自助 + 动态路由"

有两个极端都不可取。一个是"一个全能 Agent 什么都干"——prompt 越堆越长、职责混、出错难定位；另一个是"由平台开发中心化地把智能体按领域切好"——但这会卡在一个根本瓶颈：最懂某条业务的是业务线自己，不是平台开发，由平台来定义边界既不准也不可扩展。

所以我们刻意没有在开发期硬切智能体边界，而是做成自助 + 动态路由：

  • 业务线自助配置自己的智能体（连同各自的知识库），平台不替它们定义边界；
  • 后台为每个智能体的知识库静默生成一份"能力概要"，作为它"能答什么"的索引；
  • 用户提问时若没有指定某个智能体，SuperAgent 基于这些概要做匹配，把问题分发给最相关的最多 2 个智能体并行作答；
  • 再由 SuperAgent 把多方回答汇总成一份概要返回给用户。

这套设计的取舍很清楚：放弃"边界清晰"，换来"业务线自助、平台不成瓶颈"——谁懂业务谁配置，配好知识库即可被路由到，不用写复杂元数据；而边界模糊带来的"路由可能不准"，则用下面的 top-2 并行 + 汇总来兜召回。具体怎么落地，有几个关键选择：

  • 能力概要怎么来：由 LLM 读取该智能体的知识库自动生成一份自然语言的"能答什么"描述，同时知识库内容建立向量索引供后续检索。前者用于路由判断，后者用于真正作答时的 RAG——"这个智能体能干什么"既可被快速匹配、又能被精确取用，且随知识库演进，不依赖人工维护。
  • "最相关"怎么判：路由这一步不只靠向量相似度，而是把各智能体的能力概要连同用户问题一起交给 LLM 判断。相似度只能告诉你"词面接近"，LLM 才能理解"这个问题在意图上更该谁来答"——在边界模糊的场景里，语义判断比纯向量更稳。
  • 为什么是 2 个：取 1 个等于把宝全押在路由准上，边界一模糊就容易答偏；取 3 个及以上则汇总噪声和延迟明显上升。2 个是"召回兜底"和"汇总成本"之间的平衡点——既给路由留了容错，又不至于让用户等太久、读太杂。
  • 冲突怎么办：两个回答不一致时，由 SuperAgent 在汇总阶段做取舍——能判定时给综合结论，判不定时则把分歧如实呈现，而不强行抹平成一个看似确定、实则可能错的答案。

3. 上下文管理：让 Agent "知道得够、又不被淹没"

Agent 的效果上限，很大程度由"它拿到的上下文"决定。但上下文不是越多越好：

  • 太少 → 建议空泛、不贴组织实际；
  • 太多 → 噪声淹没关键信息、成本和延迟飙升、模型还会"漂"。

我们的做法是按需、分层地组装上下文，不同类型的信息走不同通道：

  • 结构化数据（组织、角色、任务等）走精确查询注入——要哪条查哪条，不靠模型猜；
  • 历史讨论做摘要与取舍，只带相关片段，而不是把整段聊天记录倒进去；
  • 知识库内容走 RAG 向量检索，只取与当前问题最相关的 top-k 片段。

三者拼起来，模型拿到的是"恰好够用"的上下文。原则只有一句：上下文是被"设计"出来的，不是把能塞的都塞进去。

4. Human-in-the-loop：把"AI 建议"变成"可审查的业务动作"

这是整个系统最关键的一道设计。我们不让模型直接改变世界，而是走一条闭环：

AI 起草
草稿
 →
人工确认
可改 / 可拒
 →
系统执行
真实落地
 →
全程追踪
可回溯

比如"把刚才群里聊的拆成任务"：AI 生成任务草稿（标题、负责人、时间），人确认或修改后才真正创建任务并进入跟踪。

这样做的代价是"不够全自动"，但收益是可控：每一个对组织有影响的动作，都有人背书、有据可查。在严肃业务里，这不是降级，而是底线。

五、可靠性工程：让 AI 在生产里可信

把 Agent 跑通是 demo，把它稳定地跑在生产里才是工程。这块我们坚持几条原则，按当前进度分成"已落地"和"在做/规划"：

已落地：

  • 工具调用有边界：每个工具的权限、可作用范围明确，避免越权——这是 Human-in-the-loop 之外的第二道约束。
  • 失败要兜底：模型超时 / 返回不可用时有降级路径，而不是把异常抛给用户。

在做 / 规划：

  • 行为可观测：对 Agent 的调用链、token / 延迟 / 成本、失败原因做记录，能回答"哪个 Agent、哪一步、为什么慢/贵/错"。
  • 效果可评测：用例集 + 评分，让"改了 prompt / 流程之后是变好还是变坏"有数据，而不是凭感觉。

我们把可观测和可评测排在后一阶段，不是因为不重要，恰恰相反——它们是从"能跑"走向"可信"的关键，也是下一步投入的重点（见第七节）。

我的判断：未来 Agent 系统的竞争力，不在"用了哪个框架"，而在"能不能让它在真实场景里可信、可控、可上线"。这也是金融背景给我的本能——出错代价高的地方，可靠性优先。

六、踩过的坑与取舍

这部分是真金白银踩出来的，比架构图更值得看：

  • 坑 1：想由平台中心化地切分智能体边界。早期倾向"由开发定义有哪些智能体、各管什么"，发现既不准、也跟不上业务变化。改成业务线自助配置 + SuperAgent 按能力概要动态路由（top-2 + 汇总）后，才既可扩展、又不让平台成为瓶颈。
  • 坑 2：上下文越喂越多，效果反而变差。一度把能拿的历史都塞进去，模型开始漂移、成本上升。改成"按需 + 摘要 + RAG"后才稳定。
  • 坑 3：A2A 协调中，智能体间的上下文 / 状态会"传丢"。多智能体接力或并行作答时，上下文在传递链路上丢失或不一致，结果就是某个智能体"答非所问"——它拿到的根本不是用户真正问的那个语境。教训是：A2A 不只是把消息转发过去，跨智能体的上下文要显式地、完整地携带和校验，不能默认"它自己会知道"。
  • 坑 4：多智能体并行，反而拖慢了整体延迟。top-2 并行作答看起来更快，但整体响应被最慢的那个智能体卡住——一个慢，用户就得全程等。这把"并行提速"的直觉打回原形：真正要做的是给汇总环节加超时与部分返回机制，让快的先到、慢的不拖死全局。
  • 取舍：克制地"不上 AI"。有些环节（如最终确认、对外动作）我们刻意保留人工，不追求全自动——可控比炫技重要。

七、结果与下一步

AIWS 目前处在内部试点 / 调研阶段：核心闭环（IM 里发起 → 多智能体协作作答 → 起草任务 → 人工确认 → 跟踪）已经跑通，正在小范围真实场景里验证、打磨。还谈不上大规模铺开，但它已经让我更确信一件事：

AI 真正的难点不在模型，而在怎么把它工程化地、可控地接进真实业务流程。

下一步我们会在可观测与评测上继续投入，让"AI 可靠性"从经验变成可度量的工程能力。
