传统会员制交易所系统设计：从产品上架到交易清算的最小 DDD 模型

设计交易所系统时，最容易犯的错误是上来就拆很多系统：用户系统、资金账户、持仓账户、交易账户、订单系统、撮合系统、清算系统、结算系统、监管系统、后台系统……每个名词都对，但第一版如果照着大交易所拆，团队很快会被复杂度压垮。

如果是一个传统会员制交易所，从最小可运行系统出发，我更建议先做“边界清楚的模块化单体”：领域边界先划清，表和事件先留好，服务不要过早拆碎。

第一版的目标不是系统名词完整，而是让主链路闭环：产品能上架，会员能交易，订单能成交，成交能清算，资产能对账，操作能审计。

一、先定一个关键前提：这是会员制交易所

传统交易所通常不是直接面对每个自然人投资者，而是面对会员、券商、做市商、清算会员等市场参与主体。因此账户模型要先分两层：

  • User：登录系统的人，负责登录、角色、权限、审计。
  • Member：市场中的交易主体，负责交易资格、资金归属、持仓归属和清算责任。

也就是说，登录用户不是资产归属主体。一个操作员登录系统代表某个会员下单，订单上应该同时记录：

operatorUserId  // 谁操作
memberId        // 谁交易、谁承担资金和持仓变化
productId       // 交易什么
orderId         // 订单

这能避免一个常见混乱：把登录账户、交易账户、资金账户、持仓账户混成一团。第一版可以不引入 tradingAccountId、cashAccountId、positionAccountId 三套业务账户 ID。资金和持仓都归属于 memberId，余额表和持仓表有自己的数据主键即可。

二、最小领域划分：4 个核心域 + 1 个支撑域

人员少、业务还在发展时，不要拆太细。一个合理的最小 DDD 划分是：

  1. 产品市场域：产品、市场、交易规则、交易时段、手续费配置。
  2. 账户资产域：用户、会员、资金余额、持仓余额、资金流水、持仓流水。
  3. 交易域：订单、撤单、前置校验、撮合、成交。
  4. 清结算账务域：清算记录、应收应付、落账、对账、日终结算。
  5. 运营监管域：产品配置、停复牌、资金/持仓调整审批、操作审计、报表查询。

这里的关键不是名字，而是职责边界：产品决定什么能交易，会员决定谁在交易，订单承载交易意图，撮合生成成交事实，清算把成交变成资产变化，流水保证可追溯。

三、一条产品完整流程怎么走

从一个产品发行上架，到行情披露、下单、撮合、清算、运营管理，主流程可以压成一条链：

产品上架
  ↓
交易规则配置
  ↓
行情基础披露
  ↓
会员账户与资产准备
  ↓
用户代表会员下单
  ↓
前置校验与资金/持仓冻结
  ↓
撮合成交
  ↓
成交回报与行情更新
  ↓
清算与落账
  ↓
对账、报表、审计、运营管理

注意，撮合只是“交易达成”的地方，清算账务才是“权利和资产变化”的依据。不要把成交表直接当成资产变动表，否则后续费用、撤销、冲正、重算和对账都会变难。

四、核心 ID 流转

第一版 ID 不要设计得太复杂。主链路里真正需要流转的是：

userId        // 操作人
memberId      // 交易主体 / 资金归属 / 持仓归属
productId     // 产品
orderId       // 订单
tradeId       // 成交
clearingId    // 清算
ledgerEntryId // 资金/持仓流水

买单校验时，用 memberId + currency 查资金余额；卖单校验时，用 memberId + productId 查持仓余额。资金余额和持仓余额有自己的数据主键，但业务上它们都属于会员。

五、第一版核心表

如果只看最小可运行系统，核心业务表可以这样收敛：

产品与规则：
- product
- product_change
- product_trading_config
- fee_config

用户与会员：
- user
- member

交易：
- order
- order_event
- trade

资产：
- cash_balance
- position_balance
- cash_ledger
- position_ledger
- cash_change_request
- position_change_request

清算与审计：
- clearing_record
- operation_audit_log

这里有几个取舍：

  • member_user 可以先不建：如果一个会员只有一个主账号，可先用 member.ownerUserId。等需要多操作员、多角色授权时再拆。
  • fee_detail 可以先不建：费用扣减结果可以进入 cash_ledger。等费用规则复杂、需要独立账单时再补费用明细。
  • order_event 建议保留：订单表只保存当前状态，订单事件保存生命周期，排查问题时非常关键。
  • clearing_record 建议保留：成交是撮合事实，清算是资产变动依据，不要混在一张表里。

六、什么时候再拆账户

第一版不用急着拆交易账户、资金账户、持仓账户。但这不代表永远不拆。等出现这些场景，再拆就有意义：

  • 一个会员有多个交易席位或交易单元。
  • 一个会员有自营、经纪、做市等多种业务账户。
  • 同一会员下有多个客户子账户。
  • 不同市场、币种、清算路径需要严格隔离资金。

在这些复杂度出现前，memberId 作为交易主体和资产归属主体，是更清楚、更少错的模型。

七、总结

传统会员制交易所的第一版系统，不应该追求“大而全”的系统名词，而应该追求主链路清楚、资产归属清楚、清算依据清楚、流水可追溯。

用户账户管登录、角色和审计；会员账户管交易主体、资金、持仓和清算责任；订单承载交易意图；撮合生成成交事实；清算把成交变成资产变化；流水保证以后能查、能审、能对账。

等业务复杂度真的上来，再拆更多账户、席位、清算路径和独立服务。早期先把模型做干净，比把系统拆得很细更重要。
