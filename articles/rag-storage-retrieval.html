RAG 的存储与检索：从原理到落地思考

做 RAG，决定成败的不是用哪个大模型，而是能否从自己的数据里检索到对的内容——而高质量的检索，靠的从来不是单一向量数据库，是按问题需求组合多种存储与检索方式。

上半篇 · 通用原理存储介质与检索方法的全景：有哪些选择、各适合什么。
下半篇 · 落地思考面对大规模文档，动手前要先想清楚的几个问题。
第一部分 · 通用篇

1这套东西到底在解什么

RAG 是 Retrieval-Augmented Generation，中文叫检索增强生成。说人话：大模型回答你之前，先去你自己的资料库里查出相关内容，拿着这些内容再作答。这样它就不靠"背书"硬答，也少编瞎话。

圈子里有句流行话——「RAG 成败在检索」。作为抓重点它没错，但照搬会误导。更准的说法是三句：

预处理 定上限

一段内容能不能被查到，在它入库那一刻就定死了。处理时漏掉的，后面再花哨的检索也捞不回来。

检索 定兑现率

这个上限能兑现多少，靠存储和检索方法的组合。工程杠杆最大、最容易翻车的就是这一段。

生成 定下限

同样的料，拼装和约束差也能答崩。但它的边际收益最低，不是主战场。

「成败在检索」真正的意思是「成败在检索质量」——它由上游处理和下游检索共同决定，检索是最集中的爆发点，不是唯一责任人。

2检索 ≠ 查向量库

大多数人一听"RAG 检索"，脑子里就是"查向量数据库"。这是个误解，而且是后面所有判断的地基。

检索，是从存储里定位并取出和问题相关的数据——跟用什么介质无关。向量检索只是其中一种，它只解决"语义相似"这一种需求而已。

举个具体的查询就明白了。假设你要查：「2023 年华东区违约的那批合同」——

SQL 查关系库、BM25 查倒排、按 key 取缓存、图遍历查关系——这些全都是检索。向量只负责其中"语义模糊"那一路。

3为什么不能只用一种

很多人会说，组合多种存储和检索是为了"高质量、高性能、高性价比"。这是表层目标。底层原因只有一句：

问题的"匹配需求"本身就是多样的，没有任何单一方式能全覆盖。

语义模糊要向量，专有名词和编号要关键词，权限和时间要结构化过滤，多跳关系要图。需求不同，方法就不同，落的介质也不同。所谓的"三高三角"，不过是这个事实在工程上的投影——你想同时把质量做高、延迟压低、成本控住，单一方案做不到，只能分工组合。

4有哪些存储介质

每种介质天生适合存某一类数据、配某一种检索。下面每张卡片按同一套看：存什么 · 怎么处理 · 怎么检索 · 成本与复杂度。成本指存储+算力的量级，复杂度指工程+运维。

对象存储 S3 / MinIO / OSS

存什么
几十 T 原始文件、扫描件、归档原件
怎么处理
几乎不处理，原样落盘
怎么检索
按路径/ID 取整个文件
成本 极低 ≈几毛/GB/月复杂度 低

存什么

怎么处理

怎么检索

这张表能反过来用：数据处理产出什么形态的"单元"（向量 / 词项 / 结构化字段 / 图三元组），就决定它落哪种介质、配哪种检索。同一段文字可以同时做成向量、进倒排、抽出字段——一份料，多路检索。

5有哪些检索方法

别按"向量 / 非向量"分，按"它解决哪种需求"分，分四类来看：召回、检索前增强、检索后处理、检索策略。

① 召回层：第一步把候选捞上来

方法解决什么需求成本复杂度
稠密向量（ANN）语义相似、模糊问法中中
稀疏 / BM25关键词、专有名词、编号低低
混合检索（hybrid + 融合）通用首选，语义+关键词互补中中
元数据过滤权限、时间、来源、类型约束低低
精确匹配（SQL / 正则）ID、编号、精确查低低
多向量后期交互（ColBERT）高精度，但存储算力都贵高高

② 检索前增强：先把"问题"问对

用户的问法常常很糙。查询改写（换种说法）、子问题分解（复杂问拆成几小问）、HyDE（先让模型脑补一段"理想答案"再拿去检索）、语义路由（先判断这个问题该走哪个库）——都是在检索发生之前提升命中率。代价是多打几次大模型。

③ 检索后处理：把捞上来的料筛精

重排（rerank）是这一类里性价比最高的一招——召回先粗捞几十条，再用一个更精的模型重新排序，质量提升明显、改造成本低。另外还有上下文压缩（裁掉无关句省 token）、父子检索（小块负责精准命中，命中后返回它所在的大块给足上下文）。

④ 检索策略：编排多路

多路召回 + 融合（几路一起捞再合并）、GraphRAG（走图做多跳）、RAPTOR（给长文档建摘要树看全局）、自适应检索（按需多轮反复检索）。这些威力大，但成本和复杂度普遍偏高，属于"有明确需求才上"。

选型心法：绝大多数场景的高性价比起点是 混合检索 + 元数据过滤 + rerank。专有名词靠 BM25、模糊问法靠向量、权限来源靠过滤、最后 rerank 精排。图检索 / RAPTOR / ColBERT 这类高代价方法，等出现了对应的明确需求再加——每多一种，都在给系统加运维负担。

第二部分 · 思考篇

6大规模数据，为什么必须先"收敛"

前面是工具箱。但真到落地，尤其当你手里是大体量的非结构化文档（动辄几十 T 的 PDF、Word、图片），第一件要想清的事不是"用哪个库"，而是：这几十 T，到底要往库里塞多少？

答案通常是：不全塞。原始数据是矿石，检索库里要的是提炼出的金属。几十 T 的原料，最终入库的可能只有几 G——中间是一个漏斗。

几十 T · 原始文件PDF / Word / 图片，含大量冗余、噪声、格式开销
▼ 解析提取正文 · OCR · 去重去噪
几十 G · 纯文本 + 结构 + 元数据原件归档进对象存储冷存，留作可追溯
▼ 切分 · 向量化 · 只留检索需要的
几 G · 可检索单元这才是在线库里真正存的精华

体积为什么能掉两三个数量级？四个来源：格式开销（PDF/图片体积大、里面纯文字很小）、去重（多版本、重复段落）、去噪（页眉页脚、水印、空白页）、只存检索所需（原件冷存，在线只放向量+元数据）。

谁来干这个漏斗：人和机器的分工

当人力不是瓶颈、但时效是硬约束时，合理的编排是一条三段级联——前段铺便宜人力把脏活粗活干掉，中段用技术扛主体量保速度，后段用专家把质量关：

人工初筛粗筛、粗分类、明显去噪（可技术辅助提速）
→
技术处理解析、OCR、去重、切分、向量化，自动化扛量
→
人工质检抽检、难例校对，卡住质量上限

不是"数据越多越好"，是"精华越纯越好"。漏斗的本事，是把对的东西留下、把噪声丢掉。

7动手前，先想清楚这 6 个问题

下面这些没有标准答案——答案取决于你的数据和场景。但每一个都会反过来决定你在第一部分该挑哪种介质、哪种方法。先把它们想清楚，再写第一行代码。

1你的数据，到底有哪几类、各占多少？

为什么关键：PDF、Word、图片之外，盘点时还要核对有没有 Excel、PPT、邮件、网页、音视频、聊天记录。其中扫描件和图片的占比，直接决定你要不要上 OCR、投入多大——而 OCR 的质量会锁死后面所有检索的上限。

2你真实的查询，长什么样？

为什么关键：是语义模糊地问、精确查编号术语、还是带权限时间过滤？这条最该先想清，因为它倒推你该配哪些介质和方法。查询都是关键词精确查，却上了一堆向量基建，就是钱花错地方。

3什么数据算"精华"，该被留下？

为什么关键：漏斗压掉什么、留下什么，全看这条标准。留太多——成本和噪声都上去、召回反而变差；留太少——该有的答案漏召回。这是一道纯数据视角的题：什么内容有检索价值。

4人工初筛 / 技术处理 / 人工质检，怎么配比？时效要多久？

为什么关键：三段的比例决定了"速度与质量"的平衡点。哪些环节靠技术全自动、哪些必须留人把关、整条管道要多久跑完一轮——这是工程节奏的总开关。

5OCR / 多模态，到什么程度才值得上？

为什么关键：扫描件和图片占比多大，才值得为 OCR 付出成本？图表里的信息是用 OCR 转文字够用，还是非得上多模态理解？质量门槛设在哪——这几个问题没答案，预算就没法估。

6你怎么知道检索"做得好不好"？

为什么关键：这条最容易被忽略，却最致命。没有评估，就没法迭代——你改了切分、换了模型，是变好还是变坏，全凭感觉。先想好用什么指标、什么样本来量它，系统才能越跑越准。

一句话收尾：做 RAG，成败在检索质量；而检索不是查一个向量库，是按"匹配需求"对多种介质取数——需求多样，所以介质和方法必须组合。而这一切的输入，是被处理过的数据：把大量原料，用人工初筛 → 技术处理 → 人工质检，提炼成又纯又对的那几 G 精华。
